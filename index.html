Case Study · ML · Healthcare

Liver Cirrhosis Prediction System

87%Accuracy
19Clinical Features
<200msInference Time

Architecture

Data Flow

  1. User fills a 19-field HTML form (age, gender, lab values, lifestyle factors)
  2. Flask API receives the POST request and validates inputs
  3. A pre-fitted sklearn Normalizer scales all numerical features
  4. RandomForestClassifier runs inference and returns class + probability
  5. Response includes prediction label, confidence %, and a contextual health tip

Model Choice Rationale

Random Forest was selected over logistic regression and SVM because:

  • Handles mixed feature types (binary flags + continuous lab values) without separate preprocessing
  • Naturally robust to outliers in medical data (e.g. extreme bilirubin values)
  • Feature importance scores give clinically interpretable insights
  • Achieves strong performance without hyperparameter sensitivity of gradient boosting

Key Trade-offs

✅ Chose

Random Forest — high accuracy, interpretable, no GPU needed, fast to serve

⚖️ Trade-off

VS XGBoost — marginally higher accuracy but slower prediction and harder to explain to doctors

✅ Chose

Normalizer over StandardScaler — more stable when feature distributions vary widely across patients

⚖️ Trade-off

No real-time retraining — model is serialised (pkl). Retraining requires `train_model.py` re-run

Case Study · LLM · Automation

LLM-Based Code Synthesis Engine

0Manual Steps
1-clickWorkflow
100%Auto-Generated Code

Architecture

Data Flow

  1. User edits question_requirements.json with problem statement and dataset path
  2. User double-clicks Run.bat → silently launches main.pyw (no terminal window)
  3. main.pyw reads both JSON configs, constructs a strict rule-based prompt
  4. Prompt sent to Gemini API → model returns executable Python code block
  5. Code is extracted, cleaned, and written directly to the project folder as a .py file
  6. Process terminates — fully headless, zero user interaction

Design Rationale

  • Gemini API chosen for generous free-tier rate limits and strong Python code generation
  • .pyw extension suppresses console window on Windows — crucial for seamless UX
  • JSON config separation decouples system config from problem definitions — users only edit question JSON
  • Policy-based prompting enforces college-exam style: minimal imports, procedural flow, no explanatory comments

Key Trade-offs

✅ Chose

Strict prompt constraints — ensures deterministic, exam-aligned output format every run

⚖️ Trade-off

Reduced creative latitude of the LLM; occasional edge-case problems may need manual prompt tweaks

✅ Chose

Background execution via .pyw — clean UX, no terminal required for end users

⚖️ Trade-off

Harder to debug — errors are silent; planned fix is logging to a local file

Case Study · Deep Learning · Freelance

Abnormal Event Detection in Surveillance

20 FPSLive Stream Handling
ConvLSTMCore Architecture
Real-timeAnomaly Alerting

Architecture

Data Flow

  1. Input: video file upload OR live webcam/IP camera feed
  2. OpenCV extracts frames; resized to 224×224 and normalised to [0,1]
  3. Sequences of 10 frames fed into the ConvLSTM Autoencoder
  4. Autoencoder reconstructs the sequence; reconstruction error computed per frame
  5. Adaptive thresholding flags frames where error > learned baseline
  6. YOLO detects objects; Haar cascades flag faces in flagged frames for privacy-aware alerts
  7. Frontend overlays anomaly score curve and highlighted frame gallery

Model Choice Rationale

  • ConvLSTM Autoencoder outperforms frame-difference methods by learning spatio-temporal normality patterns
  • Unsupervised by design — trained only on normal footage, avoiding the need for labelled anomaly datasets (rare in practice)
  • Reconstruction error as anomaly score is threshold-tunable without retraining
  • YOLO used as a secondary signal — object-level context improves alert precision for the client use case

Key Trade-offs

✅ Chose

ConvLSTM AE — trains on normal-only data, zero labelling cost, generalises across scenes

⚖️ Trade-off

VS supervised CNN — supervised model needs thousands of labelled anomaly clips the client didn't have

✅ Chose

Adaptive thresholding — adjusts per-scene rather than a hard-coded global cutoff

⚖️ Trade-off

Browser can't access RTSP directly — IP camera middleware required (documented with API endpoint)

Case Study · Blockchain · Freelance

CertChainX — Blockchain Certificate Validation

IPFSDecentralised Storage
DualVerification Modes
DockerOne-Click Deploy

Data Flow

  1. Institute logs in with pre-configured credentials stored in .env
  2. Institute fills certificate form (UID, candidate name, course, organisation)
  3. System generates a PDF certificate using ReportLab and uploads it to IPFS via the Pinata API
  4. The IPFS content hash + metadata is written to the Ethereum smart contract via Web3.py
  5. Verifier uploads a certificate PDF or enters a Certificate ID
  6. System hashes the uploaded PDF and compares against the on-chain IPFS hash — even a single-byte change causes verification failure

Design Rationale

  • Ethereum + IPFS chosen for immutability — certificate records can never be altered or deleted once on-chain
  • Content-addressed storage (IPFS) means the hash itself proves document integrity — no trusted third party needed
  • Streamlit for rapid UI prototyping with a modern glassmorphism theme — no frontend framework learning curve
  • Firebase Auth provides role-based access (Institute vs Verifier) without building custom auth from scratch
  • Docker Compose bundles Ganache + app for reproducible one-command deployment

Key Trade-offs

✅ Chose

Ethereum smart contract — tamper-proof, transparent, auditable by anyone on the network

⚖️ Trade-off

Gas costs on mainnet — development uses Ganache (local chain); production would need an L2 or consortium chain

✅ Chose

IPFS via Pinata — decentralised storage, content-addressed, no single point of failure

⚖️ Trade-off

Pinata dependency — if Pinata goes offline, pinned content persists on IPFS but re-pinning needs another provider

Case Study · AI · Computer Vision · Freelance

RecoverAI — Intelligent Lost & Found Platform

40:60Text : Image Score
ResNet50Vision Model
RBACAccess Control

Data Flow

  1. User registers with role-based access (User / Staff / Admin) — passwords hashed via Werkzeug
  2. Reporter submits a lost or found item with description, category, and optional image upload
  3. On "Find Matches", the system encodes the description using all-MiniLM-L6-v2 Sentence-Transformer
  4. Images are processed through ResNet50 (classification head removed) to extract 2048-dim feature vectors
  5. Hybrid score computed: 40% text cosine similarity + 60% image cosine similarity (text-only when no image)
  6. Matches above a 40% threshold are ranked and displayed — user can claim a match
  7. Admin reviews claims with justification, approves or rejects — status flows through Open → Matched → Claimed → Resolved

Model Choice Rationale

  • Sentence-Transformers (all-MiniLM-L6-v2) captures semantic meaning — "blue backpack" matches "navy rucksack" unlike keyword search
  • ResNet50 provides robust visual feature extraction — pre-trained on ImageNet, no custom training required
  • Hybrid 40:60 weighting prioritises visual evidence since lost item descriptions are often vague, but images are distinctive
  • Flask Blueprints architecture keeps routes modular — auth, items, claims, admin, analytics each have their own blueprint
  • Chart.js 4 provides interactive analytics — category distribution, 7-day activity timeline, resolution rates

Key Trade-offs

✅ Chose

Pre-trained models (ResNet50 + MiniLM) — zero training cost, works out of the box on first run

⚖️ Trade-off

~200MB model download on first launch — acceptable for server but heavy for edge deployment

✅ Chose

SQLite via SQLAlchemy — zero config, portable, perfect for demo; swappable to Postgres for production

⚖️ Trade-off

No real-time push notifications — in-app bell with polling; WebSocket upgrade planned for scale

SocialPulse AI — Sentiment Analytics Dashboard

TF-IDFFeature Extraction
RESTAPI Endpoint
BatchCSV Analysis

Data Flow

  1. Model is pre-trained via train_model.py — loads dataset, preprocesses text with NLTK (tokenisation, stopword removal, lemmatisation)
  2. TF-IDF vectoriser converts text into numerical features; multiple classifiers are trained and compared
  3. User enters text on the dashboard → Flask endpoint runs preprocess.py pipeline on input
  4. TF-IDF transforms the input; best-performing classifier predicts sentiment + confidence score
  5. For batch mode, user uploads a CSV → system analyses each row and returns downloadable results
  6. Model metrics page shows classifier comparison charts, confusion matrix, and classification report
  7. REST API (POST /api/predict) accepts JSON payload and returns structured prediction response

Design Rationale

  • TF-IDF + classic classifiers chosen over deep learning — faster training, lower resource usage, sufficient for sentiment classification
  • NLTK preprocessing pipeline (tokenise → stopwords → lemmatise) improves signal-to-noise ratio in text features
  • Multiple classifier comparison lets the user see which algorithm performs best on their data — transparency over black-box
  • Chart.js dashboard provides interactive visualisations without requiring a frontend framework
  • REST API endpoint enables integration with external tools, mobile apps, or automated pipelines

Key Trade-offs

✅ Chose

TF-IDF + classic ML — trains in seconds, interpretable feature weights, no GPU requirement

⚖️ Trade-off

VS BERT/Transformers — deep models would capture context better but require significant compute for training and inference

✅ Chose

Batch CSV analysis — handles bulk workloads server-side, returns downloadable results

⚖️ Trade-off

No streaming / real-time social media ingestion — would require API keys and rate-limit handling for production

Case Study · Web App · Real-Time · Freelance

NovaChat — Anonymous 1:1 Real-Time Chat

Real-timeWebSocket Chat
1:1Random Match
RenderDeploy-Ready

Data Flow

  1. User registers with email/password — passwords hashed via Werkzeug, sessions managed by Flask-Login
  2. User clicks "Find Match" → added to a server-side matching queue
  3. When two users are queued, server creates a private room and emits a matched Socket.IO event to both
  4. Messages are sent via WebSocket events — server relays to the room, stores in database for history
  5. Typing indicators fire on keypress events, debounced to avoid spam — peer sees "typing..." in real time
  6. Either user can delete their own messages, clear room history, or leave and re-enter the match queue
  7. Presence tracking shows online user count and session duration via periodic heartbeat events

Design Rationale

  • Flask-SocketIO provides WebSocket support with automatic fallback to long-polling — no separate WebSocket server needed
  • SQLAlchemy ORM with SQLite locally and PostgreSQL on Render — single codebase, zero config change for deployment
  • Server-side matching queue ensures true randomness and prevents self-matching — simple but effective algorithm
  • Gevent worker via Gunicorn handles concurrent WebSocket connections efficiently on a single Render dyno
  • Session-cookie auth over JWT — simpler for a server-rendered app, no token refresh complexity

Key Trade-offs

✅ Chose

Flask-SocketIO — unified framework for HTTP + WebSocket, single deployment unit

⚖️ Trade-off

VS Node.js/Socket.IO — Node would handle more concurrent connections, but Flask kept the stack consistent

✅ Chose

Single-worker gevent — sufficient for demo scale, avoids Redis pub/sub complexity

⚖️ Trade-off

No content moderation — reporting/blocking not yet implemented; required for any public deployment
